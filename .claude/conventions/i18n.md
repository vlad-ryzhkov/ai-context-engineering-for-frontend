Default locale: en

Translation files: `src/shared/i18n/locales/{locale}.json`

## Rules

- NEVER hardcode user-facing strings in components — use translation keys
- Translation keys: dot-notation by feature → `auth.login.title`, `users.list.empty`
- Pluralization: use ICU message format or library-specific plural rules
- Date/number formatting: use `Intl.DateTimeFormat` / `Intl.NumberFormat` or library formatters — never manual string concatenation
- RTL support: use logical CSS properties (`margin-inline-start` not `margin-left`) if RTL locales are supported

## React Example

```tsx
import { useTranslation } from 'react-i18next';

function LoginForm() {
  const { t } = useTranslation();
  return (
    <form>
      <h1>{t('auth.login.title')}</h1>
      <button type="submit">{t('auth.login.submit')}</button>
    </form>
  );
}
```

## Vue Example

```vue
<template>
  <form>
    <h1>{{ t('auth.login.title') }}</h1>
    <button type="submit">{{ t('auth.login.submit') }}</button>
  </form>
</template>

<script setup lang="ts">
import { useI18n } from 'vue-i18n';
const { t } = useI18n();
</script>
```

## When Hardcoded Strings Are Acceptable

- Internal tools with single-language requirement (document explicitly)
- Component library labels that are overridden by consumers via props
- Technical identifiers not shown to users (log messages, error codes)
